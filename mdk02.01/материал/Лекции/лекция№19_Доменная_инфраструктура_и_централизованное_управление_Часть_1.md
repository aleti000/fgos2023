## 1. Введение в службы каталогов

### 1.1. Понятие службы каталога

**Служба каталога (Directory Service)** — это специализированная база данных, предназначенная для хранения, организации и предоставления доступа к информации о сетевых ресурсах, пользователях, группах, компьютерах и других объектах в распределенной среде.

**Ключевые характеристики служб каталогов:**
- **Иерархическая структура** — данные организованы в виде дерева с четкой иерархией
- **Оптимизация для чтения** — каталоги оптимизированы для частых операций чтения и поиска
- **Строгая типизация** — каждый объект имеет определенный тип и набор атрибутов
- **Репликация** — данные могут реплицироваться между несколькими серверами
- **Безопасность** — встроенные механизмы аутентификации и авторизации

### 1.2. Назначение и функции служб каталогов

**Основные функции:**
1. **Централизованное управление** — единая точка управления всеми сетевыми ресурсами
2. **Аутентификация** — проверка подлинности пользователей и компьютеров
3. **Авторизация** — контроль доступа к ресурсам на основе политик
4. **Поиск и обнаружение** — быстрый поиск ресурсов в сети
5. **Управление конфигурацией** — централизованное применение настроек

**Примеры использования:**
- Вход пользователей в систему с едиными учетными данными
- Применение групповых политик к компьютерам и пользователям
- Публикация общих ресурсов (принтеры, папки, приложения)
- Управление правами доступа к файлам и приложениям

### 1.3. Примеры реализаций служб каталогов

**Active Directory (AD)**
- Разработчик: Microsoft
- Протоколы: LDAP, Kerberos, DNS
- Особенности: глубокая интеграция с Windows, групповые политики
- Применение: корпоративные сети на базе Windows

**OpenLDAP**
- Разработчик: OpenLDAP Project
- Протоколы: LDAP v3
- Особенности: открытая реализация, кроссплатформенность
- Применение: Unix/Linux среды, каталоги пользователей

**FreeIPA**
- Разработчик: Red Hat
- Протоколы: LDAP, Kerberos, DNS, NTP
- Особенности: комплексное решение с веб-интерфейсом
- Применение: Linux-инфраструктуры, интеграция с Windows

**Samba AD**
- Разработчик: Samba Team
- Протоколы: LDAP, Kerberos, DNS, SMB/CIFS
- Особенности: совместимость с Windows AD, открытая реализация
- Применение: смешанные среды Windows/Linux

## 2. Базовые понятия доменной инфраструктуры

### 2.1. Домен (Domain)

**Домен** — это логическая группа сетевых объектов (компьютеров, пользователей, принтеров), объединенных:
- Общей базой данных каталога
- Общими политиками безопасности
- Общим пространством имен
- Общими правилами управления

**Ключевые характеристики домена:**
- **Граница безопасности** — политики безопасности применяются на уровне домена
- **Единица репликации** — все контроллеры домена реплицируют полную копию данных
- **Пространство имен** — уникальное DNS-имя (например, company.local)
- **Административная единица** — может иметь собственных администраторов

**Функции домена:**
1. Хранение информации об объектах в иерархической структуре
2. Аутентификация пользователей и компьютеров
3. Применение политик безопасности
4. Репликация данных между контроллерами домена
5. Управление ресурсами и доступом

### 2.2. Лес (Forest)

**Лес** — это высшая структурная единица в доменной инфраструктуре, представляющая собой совокупность одного или нескольких доменов, которые:
- Имеют общую схему (schema)
- Имеют общую глобальную каталог (Global Catalog)
- Объединены транзитивными доверительными отношениями
- Имеют общую конфигурацию

**Структура леса:**
```
Лес: company.com
├── Корневой домен леса: company.com
│   ├── Пользователи: Admin, User1
│   ├── Компьютеры: DC1, Server1
│   └── Группы: Domain Admins
├── Дочерний домен: sales.company.com
│   ├── Пользователи: SalesUser1
│   ├── Компьютеры: SalesPC1
│   └── Группы: Sales Team
└── Дочерний домен: dev.company.com
    ├── Пользователи: DevUser1
    ├── Компьютеры: DevPC1
    └── Группы: Developers
```

**Особенности леса:**
- **Единая схема** — все домены в лесу используют одинаковую схему данных
- **Глобальный каталог** — содержит частичные копии всех объектов из всех доменов леса
- **Транзитивные доверия** — автоматические двунаправленные доверия между всеми доменами
- **Общая конфигурация** — настройки репликации, сайтов, служб

**Зачем нужны несколько доменов в лесу?**
1. **Административное разделение** — разные отделы могут иметь своих администраторов
2. **Разделение политик** — разные требования к паролям, блокировкам
3. **Географическое распределение** — домены для разных регионов
4. **Слияния и поглощения** — сохранение существующих структур
5. **Юридические требования** — разные юрисдикции, compliance

### 2.3. Дерево доменов (Domain Tree)

**Дерево доменов** — это иерархическая структура доменов, объединенных:
- Непрерывным пространством имен
- Транзитивными доверительными отношениями
- Общей схемой (в рамках леса)

**Пример дерева доменов:**
```
company.com (корневой домен)
├── sales.company.com (дочерний домен)
│   ├── retail.sales.company.com (внучатый домен)
│   └── wholesale.sales.company.com (внучатый домен)
├── dev.company.com (дочерний домен)
└── support.company.com (дочерний домен)
```

**Особенности именования:**
- **Непрерывное пространство имен** — каждый дочерний домен добавляет префикс к имени родителя
- **DNS-иерархия** — отражает структуру DNS-зон
- **Уникальность** — каждый домен имеет уникальное полное имя

**Отличия дерева от леса:**
| Характеристика | Дерево доменов | Лес |
|----------------|----------------|-----|
| Пространство имен | Непрерывное | Может быть прерывистым |
| Доверительные отношения | Транзитивные | Транзитивные между деревьями |
| Схема | Общая в рамках леса | Общая для всего леса |
| Количество | Может быть несколько в лесу | Всегда один в инфраструктуре |

### 2.4. Отличия между лесом, деревом и доменом

**Иерархия структур:**
```
Лес (Forest)
└── Дерево доменов (Domain Tree)
    └── Домен (Domain)
        └── Подразделение (Organizational Unit)
            └── Объекты (Users, Computers, Groups)
```

**Сравнительная таблица:**

| Параметр | Домен | Дерево | Лес |
|----------|-------|--------|-----|
| **Определение** | Логическая группа объектов | Иерархия доменов с непрерывным именем | Совокупность деревьев доменов |
| **Пространство имен** | Уникальное DNS-имя | Непрерывное (parent.child) | Может быть прерывистым |
| **Граница безопасности** | Да | Нет | Да |
| **Схема** | Наследует от леса | Общая в лесу | Единая для всего леса |
| **Доверия** | Автоматические с родительским | Транзитивные внутри дерева | Транзитивные между деревьями |
| **Администрирование** | Свои администраторы домена | Наследуют от доменов | Администраторы леса |
| **Репликация** | Полная в рамках домена | Через глобальный каталог | Через глобальный каталог |

## 3. Доверительные отношения (Trust Relationships)

### 3.1. Понятие доверительных отношений

**Доверительные отношения (Trust)** — это механизм, позволяющий пользователям из одного домена (доверяющего) получать доступ к ресурсам другого домена (доверенного) без необходимости повторной аутентификации.

**Принцип работы:**
1. Пользователь из домена A пытается получить доступ к ресурсу в домене B
2. Контроллер домена B проверяет, существует ли доверие с доменом A
3. Если доверие есть, контроллер домена A подтверждает подлинность пользователя
4. Контроллер домена B предоставляет доступ на основе подтвержденной идентичности

### 3.2. Виды доверительных отношений

#### По направлению:

**Однонаправленное доверие (One-way Trust)**
- Только один домен доверяет другому
- Пользователи из доверенного домена могут получать доступ к ресурсам доверяющего
- Пример: Домен A → Домен B (A доверяет B, но B не доверяет A)

**Двунаправленное доверие (Two-way Trust)**
- Оба домена доверяют друг другу
- Пользователи из обоих доменов могут получать взаимный доступ
- Пример: Домен A ↔ Домен B (A доверяет B и B доверяет A)

#### По транзитивности:

**Транзитивное доверие (Transitive Trust)**
- Доверие распространяется на все домены, участвующие в цепочке
- Если A доверяет B, а B доверяет C, то A автоматически доверяет C
- Пример: В дереве доменов все доверия транзитивные

**Нетранзитивное доверие (Intransitive Trust)**
- Доверие действует только между двумя конкретными доменами
- Не распространяется на другие домены
- Пример: Явные доверия между лесами

#### По способу создания:

**Неявное доверие (Implicit Trust)**
- Создается автоматически при установке домена
- Не требует ручного вмешательства администратора
- Пример: Доверие между родительским и дочерним доменом

**Явное доверие (Explicit Trust)**
- Создается администратором вручную
- Требует настройки на обоих контроллерах домена
- Пример: Кросс-лесовое доверие

### 3.3. Типы доверительных отношений в Active Directory

**1. Доверие "Родитель-Потомок" (Parent-Child Trust)**
- Создается автоматически при добавлении дочернего домена
- Двунаправленное и транзитивное
- Связывает родительский и дочерний домены в дереве

**2. Доверие между деревьями (Tree-Root Trust)**
- Создается автоматически при добавлении нового дерева в лес
- Двунаправленное и транзитивное
- Связывает корневые домены деревьев

**3. Кросс-лесовое доверие (Cross-Forest Trust)**
- Создается вручную между корневыми доменами разных лесов
- Может быть однонаправленным или двунаправленным
- Обычно нетранзитивное (требуется явное доверие для каждого домена)

**4. Явное доверие (Explicit Trust)**
- Создается вручную между любыми двумя доменами
- Может быть однонаправленным или двунаправленным
- Обычно нетранзитивное

**5. Краткосрочное доверие (Shortcut Trust)**
- Создается вручную для оптимизации маршрутов аутентификации
- Транзитивное в рамках леса
- Ускоряет процесс аутентификации в сложных иерархиях

### 3.4. Примеры доверительных отношений

**Пример 1: Простое дерево доменов**
```
company.com (корневой домен)
    ↓↑ (автоматическое, двунаправленное, транзитивное)
sales.company.com (дочерний домен)
    ↓↑ (автоматическое, двунаправленное, транзитивное)
retail.sales.company.com (внучатый домен)
```
Все домены автоматически доверяют друг другу через транзитивные отношения.

**Пример 2: Кросс-лесовое доверие**
```
Лес A: company-a.com          Лес B: company-b.com
    ↓ (однонаправленное, нетранзитивное)
    → company-b.com доверяет company-a.com
    → Пользователи из A могут получать доступ к ресурсам в B
    → Но пользователи из B не могут получать доступ к ресурсам в A
```

### 3.5. Управление доверительными отношениями

**Создание доверия (пример для Samba AD):**
```bash
# Создание явного доверия
samba-tool domain trust add DOMAIN_NAME

# Проверка существующих доверий
samba-tool domain trust list

# Удаление доверия
samba-tool domain trust remove DOMAIN_NAME
```

**Диагностика доверий:**
```bash
# Проверка работоспособности доверия
nltest /trusted_domains

# Тестирование аутентификации через доверие
nltest /server:DOMAIN_CONTROLLER /sc_query:DOMAIN_NAME
```

## 4. Роли хозяина операций (FSMO Roles)

### 4.1. Зачем нужны роли FSMO

**FSMO (Flexible Single Master Operation)** — это специальные роли, назначаемые контроллерам домена для выполнения операций, которые не могут быть выполнены одновременно на нескольких контроллерах из-за потенциальных конфликтов.

**Проблема многомастер-репликации:**
В доменной инфраструктуре все контроллеры домена равноправны и могут принимать изменения (многомастер-модель). Однако некоторые операции требуют уникальности:
- Изменение схемы домена
- Добавление нового домена в лес
- Изменение относительных идентификаторов (RID)
- Обновление времени для синхронизации
- Изменение инфраструктуры

**Решение:** Назначить определенные роли конкретным контроллерам домена, которые будут единолично выполнять критические операции.

### 4.2. Лесовые роли (Forest-wide Roles)

Эти роли существуют в единственном экземпляре во всем лесу.

#### **Schema Master (Мастер схемы)**

**Назначение:**
- Единственный контроллер домена, который может вносить изменения в схему Active Directory
- Схема определяет типы объектов и их атрибуты

**Функции:**
1. Внесение изменений в схему (добавление классов, атрибутов)
2. Обновление схемы при установке новых версий ПО
3. Применение расширений схемы от сторонних приложений

**Последствия потери:**
- Невозможность изменять схему
- Невозможность устанавливать приложения, требующие расширения схемы
- Существующие объекты продолжают работать

**Пример изменения схемы:**
```bash
# Просмотр текущей схемы
samba-tool schema show

# Добавление нового атрибута (требует роли Schema Master)
samba-tool schema addattribute --name=newAttribute --type=string
```

#### **Domain Naming Master (Мастер имен доменов)**

**Назначение:**
- Единственный контроллер домена, который может добавлять или удалять домены в лесу
- Управляет добавлением и удалением ссылок на приложения в разделах приложений

**Функции:**
1. Добавление новых доменов в лес
2. Удаление доменов из леса
3. Добавление/удаление разделов приложений
4. Управление именами доменов для обеспечения уникальности

**Последствия потери:**
- Невозможность добавлять новые домены
- Невозможность удалять существующие домены
- Существующие домены продолжают работать

**Пример:**
При попытке добавить новый домен без доступного Domain Naming Master:
```
Ошибка: Не удалось связаться с мастером имен доменов.
Невозможно добавить новый домен в лес.
```

### 4.3. Доменные роли (Domain-wide Roles)

Эти роли существуют в единственном экземпляре в каждом домене.

#### **PDC Emulator (Эмулятор основного контроллера домена)**

**Назначение:**
- Обеспечивает обратную совместимость с Windows NT 4.0
- Выполняет критические функции синхронизации времени и обработки изменений

**Функции:**
1. **Синхронизация времени** — источник точного времени for all контроллеров домена
2. **Обработка изменений паролей** — немедленная репликация изменений паролей
3. **Аутентификация** — обработка неудачных попыток аутентификации (блокировка учетной записи)
4. **Совместимость с NT** — эмуляция PDC для устаревших клиентов
5. **Групповые политики** — предпочтительный источник для редактирования GPO

**Последствия потери:**
- Проблемы с синхронизацией времени
- Задержки в репликации изменений паролей
- Проблемы с блокировкой учетных записей
- Невозможность редактирования групповых политик

**Пример важности PDC Emulator:**
```bash
# Пользователь меняет пароль на контроллере домена A
# PDC Emulator немедленно получает это изменение
# Другие контроллеры домена реплицируют изменение с PDC Emulator
# Это обеспечивает быструю доступность нового пароля
```

#### **RID Master (Мастер относительных идентификаторов)**

**Назначение:**
- Распределяет пулы относительных идентификаторов (RID) между контроллерами домена
- RID — это уникальная часть SID (Security Identifier)

**Структура SID:**
```
S-1-5-21-3623811015-3361044348-30300820-1013
│ │ │ │                       │               │
│ │ │ │                       │               └── RID (относительный идентификатор)
│ │ │ │                       └────────────────── Доменный идентификатор
│ │ │ └────────────────────────────────────────── Идентификатор NT Authority
│ │ └──────────────────────────────────────────── Идентификатор безопасности
│ └────────────────────────────────────────────── Номер версии
└──────────────────────────────────────────────── Идентификатор SIDs
```

**Функции:**
1. Выделение пулов RID контроллерам домена (обычно по 500 RID)
2. Отслеживание использования RID
3. Предотвращение дублирования RID

**Последствия потери:**
- Контроллеры домена не могут создавать новые объекты (пользователей, группы, компьютеры)
- При исчерпании текущего пула RID новые объекты не создаются
- Существующие объекты продолжают работать

**Пример:**
```bash
# Проверка доступных RID на контроллере домена
dsquery * "CN=RID Manager$,CN=System,DC=domain,DC=com" -attr ridAvailable

# Если доступно 0 RID, контроллер домена запрашивает новый пул у RID Master
```

#### **Infrastructure Master (Мастер инфраструктуры)**

**Назначение:**
- Обновляет ссылки на объекты из других доменов
- Поддерживает актуальность cross-domain ссылок

**Функции:**
1. Обновление ссылок на объекты из других доменов
2. Обновление членства в универсальных группах
3. Поддержание актуальности ссылок при переименовании объектов

**Особенности:**
- **Не требуется**, если все контроллеры домена являются серверами глобального каталога
- **Критичен** в многодоменных средах без полного глобального каталога

**Последствия потери:**
- Устаревание ссылок на объекты из других доменов
- Проблемы с членством в универсальных группах
- Невозможность корректного отображения объектов из других доменов

**Пример:**
```
Пользователь из домена A добавляется в группу в домене B.
Infrastructure Master в домене B обновляет ссылку на пользователя из домена A.
Если Infrastructure Master недоступен, ссылка может устареть.
```

### 4.4. Распределение ролей FSMO

**Рекомендации по размещению:**

| Роль | Рекомендуемое размещение |
|------|--------------------------|
| Schema Master | Корневой домен леса, надежный контроллер |
| Domain Naming Master | Корневой домен леса, рядом со Schema Master |
| PDC Emulator | Мощный контроллер с хорошим сетевым подключением |
| RID Master | Надежный контроллер, доступный всем другим |
| Infrastructure Master | Не на сервере глобального каталога (если не все контроллеры — GC) |

**Пример распределения в небольшом домене:**
```
Контроллер домена 1 (основной):
- Schema Master
- Domain Naming Master
- PDC Emulator
- RID Master

Контроллер домена 2 (дополнительный):
- Infrastructure Master
```

### 4.5. Перемещение и захват ролей

**Плановое перемещение ролей:**
```bash
# Перемещение роли PDC Emulator на другой контроллер
samba-tool fsmo set --role=pdc --server=new_dc_name

# Перемещение всех ролей на указанный контроллер
samba-tool fsmo set --all --server=new_dc_name
```

**Аварийный захват ролей (если контроллер с ролью недоступен):**
```bash
# Захват роли Schema Master
samba-tool fsmo seize --role=schema

# Захват всех ролей
samba-tool fsmo seize --all
```

**Важно:** Захват ролей следует использовать только в аварийных ситуациях, так как это может привести к рассинхронизации, если оригинальный контроллер снова станет доступен.

### 4.6. Проверка ролей FSMO

**Просмотр текущих владельцев ролей:**
```bash
# В Samba AD
samba-tool fsmo show

# В Windows AD
netdom query fsmo
```

**Пример вывода:**
```
Schema Master: DC1.company.com
Domain Naming Master: DC1.company.com
PDC Emulator: DC2.company.com
RID Master: DC1.company.com
Infrastructure Master: DC3.company.com
```

## 5. Типы доменов и их реализации

### 5.1. OpenLDAP

**Описание:**
OpenLDAP — это открытая реализация протокола LDAP (Lightweight Directory Access Protocol), предназначенная для создания и управления службами каталогов.

**Архитектура:**
```
OpenLDAP Server
├── База данных (LDAP Directory)
├── Служба каталогов (slapd)
├── Утилиты управления
└── Клиентские библиотеки
```

**Особенности:**
- **Протокол:** LDAP v3
- **База данных:** Может использовать различные бэкенды (Berkeley DB, SQL, LDIF)
- **Схема:** Гибкая, расширяемая
- **Репликация:** Multi-master или master-slave
- **Безопасность:** TLS/SSL, SASL

**Преимущества:**
- Открытый исходный код
- Кроссплатформенность (Linux, Unix, Windows)
- Гибкость и настраиваемость
- Поддержка различных бэкендов баз данных
- Зрелая и стабильная реализация

**Недостатки:**
- Нет встроенной поддержки Kerberos (требует отдельной настройки)
- Нет групповых политик
- Сложность настройки для начинающих
- Ограниченная интеграция с Windows

**Типичные сценарии использования:**
- Каталоги пользователей в Unix/Linux средах
- Централизованная аутентификация
- Хранение контактной информации
- Интеграция с веб-приложениями

**Пример конфигурации:**
```bash
# Установка OpenLDAP в ALT Linux
apt-get install slapd ldap-utils

# Настройка базового DN
slapadd -l base.ldif

# Проверка работы
ldapsearch -x -b "dc=company,dc=com" "(objectclass=*)"
```

### 5.2. Samba AD

**Описание:**
Samba AD — это реализация Active Directory с открытым исходным кодом, обеспечивающая полную совместимость с Windows Active Directory.

**Архитектура:**
```
Samba AD Domain Controller
├── LDAP сервер (для хранения объектов)
├── Kerberos KDC (для аутентификации)
├── DNS сервер (для разрешения имен)
├── SMB/CIFS сервер (для доступа к файлам)
└── Winbind (для интеграции с Unix-системами)
```

**Особенности:**
- **Протоколы:** LDAP, Kerberos, DNS, SMB/CIFS
- **Совместимость:** Полная совместимость с Windows AD
- **Роли FSMO:** Поддерживает все роли FSMO
- **Групповые политики:** Поддержка базовых GPO
- **Доверия:** Поддержка доверий с Windows AD

**Преимущества:**
- Полная совместимость с Windows AD
- Открытый исходный код
- Поддержка смешанных сред Windows/Linux
- Встроенные DNS и Kerberos
- Поддержка групповых политик
- Активное сообщество

**Недостатки:**
- Меньше функций, чем в Windows AD
- Ограниченная поддержка сложных GPO
- Требует тщательной настройки
- Меньше документации, чем для Windows AD

**Типичные сценарии использования:**
- Смешанные среды Windows/Linux
- Замена Windows Server в небольших организациях
- Интеграция Linux-клиентов в Windows AD
- Создание доменной инфраструктуры на базе Linux

**Пример установки контроллера домена:**
```bash
# Установка пакетов Samba AD
apt-get install samba-dc

# Provision домена
samba-tool domain provision --use-rfc2307 --interactive

# Запуск службы
systemctl enable --now samba-ad-dc

# Проверка работы
samba-tool domain info
```

### 5.3. FreeIPA

**Описание:**
FreeIPA (Identity, Policy, Audit) — это комплексное решение для управления идентификацией, политиками и аудитом, разработанное Red Hat.

**Архитектура:**
```
FreeIPA Server
├── 389 Directory Server (LDAP)
├── Kerberos KDC
├── DNS сервер (BIND)
├── NTP сервер
├── PKI (Certificate Authority)
├── Веб-интерфейс
└── Система управления политиками
```

**Особенности:**
- **Интеграция:** Все компоненты интегрированы "из коробки"
- **Управление:** Веб-интерфейс и CLI
- **Политики:** HBAC (Host-Based Access Control), SELinux политики
- **Аудит:** Централизованное логирование
- **Доверия:** Поддержка доверий с Windows AD

**Преимущества:**
- Комплексное решение "всё в одном"
- Удобный веб-интерфейс
- Отличная интеграция с Linux-системами
- Встроенный центр сертификации
- Поддержка двухфакторной аутентификации
- Активное развитие

**Недостатки:**
- Ориентирован в первую очередь на Linux-среды
- Ограниченная поддержка Windows-клиентов
- Требует значительных ресурсов
- Сложность масштабирования

**Типичные сценарии использования:**
- Linux-инфраструктуры
- Научные и образовательные учреждения
- Организации, требующие строгого контроля доступа
- Среды с высокими требованиями к безопасности

**Пример установки FreeIPA:**
```bash
# Установка пакетов
apt-get install freeipa-server freeipa-client

# Настройка сервера
ipa-server-install --setup-dns

# Добавление клиента
ipa-client-install
```

### 5.4. Сравнительная характеристика

| Характеристика | OpenLDAP | Samba AD | FreeIPA |
|----------------|----------|----------|---------|
| **Основное назначение** | Служба каталогов | Active Directory | Комплексное управление идентификацией |
| **Протоколы** | LDAP | LDAP, Kerberos, DNS, SMB | LDAP, Kerberos, DNS, NTP, PKI |
| **Совместимость with Windows** | Ограниченная | Полная | Через доверия |
| **Групповые политики** | Нет | Базовые | HBAC, SELinux |
| **Веб-интерфейс** | Сторонние решения | Ограниченный | Полнофункциональный |
| **Центр сертификации** | Нет | Нет | Да |
| **Сложность настройки** | Высокая | Средняя | Средняя |
| **Требования к ресурсам** | Низкие | Средние | Высокие |
| **Поддержка сообщества** | Отличная | Отличная | Отличная |
| **Лицензия** | Открытая | Открытая | Открытая |

## 6. Заключение

### 6.1. Ключевые понятия

1. **Служба каталога** — специализированная база данных для хранения информации о сетевых ресурсах
2. **Домен** — логическая группа объектов с общей базой данных и политиками
3. **Лес** — высшая структурная единица, объединяющая несколько доменов
4. **Дерево доменов** — иерархия доменов с непрерывным пространством имен
5. **Доверительные отношения** — механизм взаимного доступа между доменами
6. **Роли FSMO** — специальные роли для выполнения уникальных операций
7. **Контроллер домена** — сервер, хранящий копию базы данных домена
