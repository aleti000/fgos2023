## Введение

### Понятие DNS и его роль в сетевой инфраструктуре

Domain Name System (DNS) — это распределенная система, предназначенная для преобразования доменных имен в IP-адреса и наоборот. DNS играет ключевую роль в функционировании Интернета и корпоративных сетей, обеспечивая удобное для пользователей именование сетевых ресурсов.

**Основные функции DNS:**
- Преобразование доменных имен в IP-адреса (прямое разрешение)
- Преобразование IP-адресов в доменные имена (обратное разрешение)
- Хранение и распространение информации о сетевых ресурсах
- Обеспечение отказоустойчивости и распределения нагрузки

**Преимущества использования DNS:**
- Удобство для пользователей (запоминание имен вместо чисел)
- Централизованное управление сетевыми ресурсами
- Возможность быстрого изменения IP-адресов без изменения имен
- Поддержка балансировки нагрузки и отказоустойчивости

### История развития DNS

**Предшественники DNS:**
- **HOSTS.TXT** — централизованный файл с соответствиями имен и адресов
- **Проблемы HOSTS.TXT:**
  - Необходимость синхронизации на всех узлах
  - Ограниченность масштабирования
  - Зависимость от центрального узла

**Развитие DNS:**
- **1983 год** — RFC 882 и RFC 883 определили основы DNS
- **1987 год** — RFC 1034 и RFC 1035 стали стандартами DNS
- **1990-е годы** — массовое внедрение DNS в Интернете
- **2000-е годы** — развитие DNSSEC и расширенных возможностей

**Современное состояние:**
- DNS стал основой функционирования Интернета
- Поддержка IPv6 и новых типов записей
- Развитие систем безопасности DNS
- Интеграция с облачными технологиями

### Преимущества использования DNS

**Для пользователей:**
- Простота запоминания имен
- Возможность использования понятных имен для ресурсов
- Автоматическое разрешение имен без вмешательства пользователя

**Для администраторов:**
- Централизованное управление сетевыми ресурсами
- Возможность быстрого изменения конфигурации
- Поддержка отказоустойчивости и балансировки нагрузки
- Упрощение миграции и переименования ресурсов

**Для сети в целом:**
- Распределенная архитектура повышает отказоустойчивость
- Кэширование уменьшает нагрузку на сеть
- Поддержка различных типов ресурсов и служб

## Теоретическая часть

### Принципы работы DNS

#### Архитектура клиент-серверной модели

**Компоненты DNS-инфраструктуры:**

1. **DNS-клиент (резолвер)** — приложение, запрашивающее разрешение имен
2. **DNS-сервер** — устройство, отвечающее на DNS-запросы
3. **Корневые серверы** — серверы верхнего уровня иерархии
4. **TLD-серверы** — серверы доменов верхнего уровня (.com, .org, .ru и т.д.)
5. **Авторитетные серверы** — серверы, хранящие информацию о конкретных доменах

**Типы DNS-серверов:**
- **Рекурсивные серверы** — выполняют полный процесс разрешения имени
- **Итеративные серверы** — возвращают ссылки на другие серверы
- **Кэширующие серверы** — хранят результаты запросов для ускорения доступа
- **Авторитетные серверы** — хранят информацию о конкретных зонах

#### Процесс разрешения доменных имен

**1. Прямое разрешение (имя → IP-адрес):**

```bash
# Процесс разрешения имени example.com
1. Клиент → Локальный DNS-сервер
2. Локальный сервер → Корневой сервер
3. Корневой сервер → TLD-сервер (.com)
4. TLD-сервер → Авторитетный сервер (example.com)
5. Авторитетный сервер → IP-адрес
6. Результат возвращается клиенту
```

**2. Обратное разрешение (IP-адрес → имя):**

```bash
# Процесс обратного разрешения
1. IP-адрес преобразуется в обратное имя
2. Запрос направляется в зону in-addr.arpa
3. Процесс аналогичен прямому разрешению
4. Результат: IP-адрес ↔ Доменное имя
```

**3. Типы DNS-запросов:**
- **Рекурсивный запрос** — сервер должен выполнить полный поиск
- **Итеративный запрос** — сервер возвращает лучший доступный ответ
- **Непосредственный запрос** — запрос конкретному серверу

#### Типы DNS-записей и их назначение

**Основные типы DNS-записей:**

| Тип записи | Назначение | Пример |
|------------|------------|---------|
| A | IPv4 адрес | example.com → 192.168.1.1 |
| AAAA | IPv6 адрес | example.com → 2001:db8::1 |
| CNAME | Псевдоним | www → example.com |
| MX | Почтовый сервер | example.com → mail.example.com |
| NS | DNS-сервер зоны | example.com → ns1.example.com |
| SOA | Начало полномочий зоны | Информация о зоне |
| PTR | Обратное разрешение | 1.1.168.192.in-addr.arpa → example.com |
| TXT | Текстовая информация | SPF, DKIM записи |
| SRV | Служебные записи | _sip._tcp.example.com |
| SPF | Защита от спама | Указание разрешенных серверов |
| DKIM | Аутентификация почты | Криптографическая подпись |

**Примеры использования записей:**

```bash
# A запись
example.com. IN A 192.168.1.1

# MX запись с приоритетом
example.com. IN MX 10 mail.example.com.

# CNAME запись
www.example.com. IN CNAME example.com.

# TXT запись для SPF
example.com. IN TXT "v=spf1 mx a -all"
```

#### Зоны DNS и их виды

**1. Первичные (мастер) зоны:**
- Содержат оригинальные записи зоны
- Только один первичный сервер на зону
- Записи хранятся в локальных файлах

**2. Вторичные (слейв) зоны:**
- Содержат копии первичной зоны
- Обеспечивают отказоустойчивость
- Регулярно синхронизируются с первичным сервером

**3. Обратные зоны:**
- Обеспечивают обратное разрешение имен
- Используются для преобразования IP-адресов в имена
- Располагаются в доменах in-addr.arpa (IPv4) и ip6.arpa (IPv6)

**4. Интегрированные зоны:**
- Записи хранятся в Active Directory
- Автоматическая репликация между серверами
- Используется в Windows-средах

### DNS-серверы и их типы

#### Основные функции DNS-сервера

**1. Разрешение имен:**
- Обработка DNS-запросов от клиентов
- Выполнение рекурсивных и итеративных запросов
- Возврат результатов клиентам

**2. Кэширование:**
- Хранение результатов запросов
- Уменьшение времени ответа
- Снижение нагрузки на сеть

**3. Управление зонами:**
- Хранение и обслуживание DNS-записей
- Обеспечение целостности данных
- Репликация между серверами

**4. Безопасность:**
- Фильтрация запросов
- Защита от атак
- Аутентификация и авторизация

#### Типы DNS-серверов

**1. Авторитетные серверы:**
- Хранят информацию о конкретных зонах
- Отвечают на запросы об этих зонах
- Могут быть первичными или вторичными

**2. Рекурсивные серверы:**
- Выполняют полный процесс разрешения
- Обращаются к другим серверам при необходимости
- Кэшируют результаты запросов

**3. Кэширующие серверы:**
- Хранят результаты предыдущих запросов
- Ускоряют разрешение часто используемых имен
- Не хранят информацию о зонах

**4. Передающие серверы (forwarders):**
- Перенаправляют запросы другим серверам
- Используются для оптимизации трафика
- Позволяют централизовать выход в Интернет

#### Иерархия DNS

**1. Корневые серверы:**
- Находятся на вершине иерархии
- Обрабатывают запросы к корневой зоне
- Перенаправляют запросы к TLD-серверам

**2. TLD-серверы:**
- Обслуживают домены верхнего уровня
- Хранят информацию о доменах второго уровня
- Перенаправляют запросы к авторитетным серверам

**3. Авторитетные серверы:**
- Обслуживают конкретные домены
- Хранят информацию о поддоменах
- Отвечают на запросы об этих доменах

**4. Локальные серверы:**
- Обслуживают локальные сети
- Кэшируют результаты запросов
- Обеспечивают доступ к внешним ресурсам

#### Взаимодействие между DNS-серверами

**1. Зональная передача (zone transfer):**
- Передача информации о зоне между серверами
- Используется для синхронизации вторичных серверов
- Может быть полной (AXFR) или частичной (IXFR)

**2. Уведомления о изменениях:**
- Первичный сервер уведомляет вторичные об изменениях
- Вторичные серверы инициируют зональную передачу
- Обеспечивает актуальность данных

**3. Рекурсивные запросы:**
- Серверы могут запрашивать информацию у других серверов
- Создается цепочка запросов до нахождения ответа
- Результат кэшируется для ускорения последующих запросов

### Безопасность DNS

#### DNS-атаки и методы защиты

**1. DNS-спуфинг:**
- Подмена DNS-ответов
- Перенаправление на фальшивые сайты
- Методы защиты: DNSSEC, проверка подлинности

**2. DNS-кэширование отравления:**
- Загрузка ложной информации в кэш
- Влияние на множество пользователей
- Методы защиты: случайные порты, проверка ответов

**3. DDoS-атаки на DNS:**
- Перегрузка DNS-серверов запросами
- Нарушение работы DNS-инфраструктуры
- Методы защиты: фильтрация, распределение нагрузки

**4. DNS-туннелирование:**
- Использование DNS для передачи данных
- Обход防火墙 и систем фильтрации
- Методы защиты: мониторинг аномалий, ограничение размера запросов

#### DNSSEC

**Принципы DNSSEC:**
- Цифровая подпись DNS-записей
- Проверка подлинности данных
- Защита от подмены и модификации

**Компоненты DNSSEC:**
- **DS-записи** — дайджесты ключей поддоменов
- **DNSKEY** — публичные ключи для проверки подписей
- **RRSIG** — цифровые подписи для DNS-записей
- **NSEC/NSEC3** — подтверждение отсутствия записей

**Преимущества DNSSEC:**
- Защита от подмены DNS-данных
- Обеспечение целостности информации
- Повышение доверия к DNS

**Ограничения DNSSEC:**
- Сложность настройки и управления
- Увеличение размера DNS-пакетов
- Не защищает от DDoS-атак

#### Взаимодействие с другими сетевыми службами

**1. Интеграция с DHCP:**
- Автоматическое обновление DNS-записей
- Динамическое добавление и удаление записей
- Синхронизация информации о хостах

**2. Интеграция с Active Directory:**
- Хранение DNS-записей в каталоге
- Автоматическая репликация
- Интеграция с системой аутентификации

**3. Взаимодействие с почтовыми серверами:**
- MX-записи для маршрутизации почты
- SPF, DKIM, DMARC для защиты от спама
- Проверка подлинности отправителей

### Особенности DNS-сервера bind в ALT Linux

#### Архитектура bind в ALT Linux 2.4 Master

В ALT Linux 2.4 Master DNS-сервер bind (демон named) работает в специальном chroot-окружении для повышения безопасности. Это важная особенность, которую необходимо учитывать при настройке.

**Особенности chroot-окружения:**
- Named запускается в каталоге `/var/lib/bind` как корневом каталоге
- Все пути в конфигурации указываются относительно этого каталога
- Главный файл настроек `/etc/named.conf` является символьной ссылкой
- Конфигурационные файлы находятся в `/var/lib/bind/etc/`
- Зоны хранятся в `/var/lib/bind/zone/`

**Структура каталогов bind в ALT Linux:**
```bash
/var/lib/bind/                    # Корень chroot-окружения
├── etc/                         # Конфигурационные файлы
│   ├── named.conf              # Главный конфигурационный файл
│   ├── options.conf            # Настройки сервера
│   ├── rfc1912.conf            # Стандартные зоны
│   ├── local.conf              # Локальные зоны
│   └── ...
├── zone/                       # Файлы DNS-зон
│   ├── empty                   # Пустая зона
│   ├── internal.domain.net     # Пример прямой зоны
│   └── 12.11.10.in-addr.arpa   # Пример обратной зоны
└── dev/                        # Системные устройства
```

```bash
ls -la /var/lib/bind/

# Проверка состояния службы

```

#### Настройка кэширующего DNS-сервера

**1. Редактирование конфигурации options.conf:**
```bash
# Редактирование файла настроек
nano /var/lib/bind/etc/options.conf
```

**2. Раскомментировать и заполнить раздел forwarders:**
```conf
# /var/lib/bind/etc/options.conf
options {
    directory "/var/lib/bind/zone";
    
    # Настройка вышестоящих DNS-серверов
    forwarders {
        8.8.8.8;      # Google DNS
        1.1.1.1;      # Cloudflare DNS
    };
    
    # Ограничение доступа к серверу
    allow-query { localnets; };
    allow-recursion { localnets; };
    
    # Другие настройки
    listen-on { any; };
    listen-on-v6 { any; };
};
```

**3. Проверка настроек доступа:**
```bash
# Проверка настроек allow-query и allow-recursion
grep -A 5 "allow-" /var/lib/bind/etc/options.conf
```

#### Настройка первичного DNS-сервера

**1. Создание локальной зоны:**
```bash
# Редактирование файла локальных зон
nano /var/lib/bind/etc/local.conf
```

**2. Добавление описания зоны:**
```conf
# /var/lib/bind/etc/local.conf
zone "internal.domain.net" {
    type master;
    file "internal.domain.net";
};

zone "12.11.10.in-addr.arpa" {
    type master;
    file "12.11.10.in-addr.arpa";
};
```

**3. Создание прямой зоны:**
```bash
# Создание файла прямой зоны
cat > /var/lib/bind/zone/internal.domain.net << 'EOF'
$TTL    1D
@       IN      SOA     server.internal.domain.net. root.server.internal.domain.net. (
                                2004082202      ; serial
                                12H             ; refresh
                                1H              ; retry
                                1W              ; expire
                                1H              ; ncache
                        )
        IN      NS      server.internal.domain.net.
        IN      MX 10   server.internal.domain.net.
server          IN      A       10.10.10.1
www             IN      CNAME   server.internal.domain.net.
mail            IN      CNAME   server.internal.domain.net.
jack            IN      A       10.10.10.100
jill            IN      A       10.10.10.101
EOF
```

**4. Создание обратной зоны:**
```bash
# Создание файла обратной зоны
cat > /var/lib/bind/zone/12.11.10.in-addr.arpa << 'EOF'
$TTL    1D
@       IN      SOA     server.internal.domain.net. root.server.internal.domain.net. (
                                2004082201      ; serial
                                12H             ; refresh
                                1H              ; retry
                                1W              ; expire
                                1H              ; ncache
                        )
        IN      NS      server.internal.domain.net.
0               IN      PTR     internal.domain.net.
1               IN      PTR     server.internal.domain.net.
100             IN      PTR     jack.internal.domain.net.
101             IN      PTR     jill.internal.domain.net.
EOF
```

### Альтернатива: Настройка pdnsd (кеширующий DNS-сервер)

#### Установка и настройка pdnsd

**1. Установка pdnsd:**
```bash
# Установка pdnsd
apt-get install pdnsd -y
```

**2. Настройка pdnsd:**
```bash
# Редактирование конфигурации pdnsd
nano /etc/pdnsd.conf
```

**3. Настройка вышестоящих DNS-серверов:**
```conf
# /etc/pdnsd.conf
global {
    perm_cache=1024;
    cache_dir="/var/cache/pdnsd";
    pid_file = /var/run/pdnsd.pid;
    run_as="pdnsd";
    server_ip = 127.0.0.1;  # Use eth0 here if you want to allow other
                            # machines on your network to query pdnsd.
    status_ctl = on;
    paranoid=on;       # This option reduces the chance of cache poisoning
                       # but may make pdnsd less efficient, unfortunately.
    query_method=udp_tcp;
    max_ttl=1w;        # One week.
    neg_domain_pol=on;
    udpbufsize=1024;   # Upper limit on the size of UDP messages.
}

server {
    label= "provider";
    ip = 8.8.8.8, 1.1.1.1;  # Google DNS и Cloudflare DNS
    timeout=10;
    interval=10;     # Test every 10 seconds.
    uptest=ping;
    ping_timeout=300;
    purge_cache=off;
    exclude=.localdomain;
    policy=included;
    preset=off;
}
```

## Глоссарий терминов

- **DNS (Domain Name System)** — система доменных имен
- **DNS-сервер** — сервер, осуществляющий разрешение доменных имен
- **DNS-клиент (резолвер)** — приложение, запрашивающее разрешение имен
- **DNS-зона** — часть DNS-пространства, управляемая определенным сервером
- **A-запись** — запись, сопоставляющая доменное имя с IPv4-адресом
- **AAAA-запись** — запись, сопоставляющая доменное имя с IPv6-адресом
- **CNAME-запись** — запись псевдонима для доменного имени
- **MX-запись** — запись почтового сервера
- **NS-запись** — запись DNS-сервера зоны
- **SOA-запись** — запись начала полномочий зоны
- **PTR-запись** — запись для обратного разрешения имен
- **DNSSEC** — система безопасности DNS
- **Рекурсивный запрос** — запрос, при котором сервер должен выполнить полный поиск
- **Итеративный запрос** — запрос, при котором сервер возвращает лучший доступный ответ
- **Зональная передача** — процесс передачи информации о зоне между серверами
