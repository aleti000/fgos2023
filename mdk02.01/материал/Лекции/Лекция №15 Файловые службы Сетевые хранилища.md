## Теоретические основы файловых служб и сетевого хранения

### Концепция распределенных файловых систем

Распределенные файловые системы представляют собой фундаментальную технологию в современных информационных системах, обеспечивающую централизованное хранение и доступ к данным через сеть. В отличие от локальных файловых систем, распределенные системы позволяют нескольким пользователям и приложениям одновременно получать доступ к файлам, независимо от их физического расположения.

**Принципы сетевого хранения данных**

Сетевое хранение данных основано на нескольких ключевых принципах:

1. **Абстракция физического хранения** - пользователи и приложения работают с логическими ресурсами, не зная об их физическом расположении
2. **Прозрачный доступ** - сетевые файлы доступны так же, как и локальные
3. **Консистентность данных** - обеспечивается целостность и согласованность данных при одновременном доступе
4. **Масштабируемость** - возможность расширения хранилища без остановки работы системы

**Архитектура клиент-серверных взаимодействий**

Клиент-серверная архитектура в сетевых файловых системах включает:

- **Файловый сервер** - централизованное хранилище данных с управлением доступом
- **Клиентские системы** - устройства, получающие доступ к удаленным файлам
- **Сетевой протокол** - средство передачи запросов и данных между клиентом и сервером
- **Система аутентификации** - механизм проверки подлинности пользователей

### Архитектура файловых серверов

**Модель управления доступом к файлам**

Файловые серверы реализуют многоуровневую модель управления доступом:

1. **Уровень аутентификации** - проверка подлинности пользователя
2. **Уровень авторизации** - определение прав доступа пользователя
3. **Уровень разрешений** - конкретные права на операции с файлами и каталогами
4. **Уровень аудита** - ведение журналов доступа и операций

**Протокольные уровни и взаимодействие**

Сетевые файловые системы используют многоуровневую протокольную архитектуру:

- **Транспортный уровень** - TCP/IP, UDP
- **Сеансовый уровень** - установление и поддержание соединений
- **Прикладной уровень** - специализированные протоколы доступа к файлам

**Безопасность и аутентификация в сетевых файлах**

Безопасность сетевых файловых систем обеспечивается несколькими механизмами:

- **Шифрование данных** - защита от несанкционированного доступа при передаче
- **Аутентификация пользователей** - проверка подлинности через различные методы
- **Контроль доступа** - разграничение прав на основе ролей и групп
- **Аудит и мониторинг** - ведение журналов для анализа и расследования инцидентов

## Часть 1: NFS (Network File System)

### Теоретические основы NFS

**История и развитие NFS**

Network File System (NFS) был разработан компанией Sun Microsystems в 1984 году как протокол для доступа к файлам по сети. История развития NFS включает несколько ключевых этапов:

- **NFSv2 (1989)** - первая широко распространенная версия, использующая UDP
- **NFSv3 (1995)** - добавлена поддержка TCP, улучшена производительность
- **NFSv4 (2000)** - кардинальные изменения: безопасность, состояние, Unicode
- **NFSv4.1 (2010)** - поддержка параллельного доступа (pNFS)
- **NFSv4.2 (2016)** - улучшения производительности и безопасности

**Архитектура клиент-серверного взаимодействия**

Архитектура NFS основана на модели "запрос-ответ":

1. **Клиент NFS** - инициирует запросы к серверу
2. **Сервер NFS** - обрабатывает запросы и возвращает результаты
3. **RPC (Remote Procedure Call)** - механизм удаленного вызова процедур
4. **Portmapper** - служба определения портов для сервисов NFS

**Принципы работы и особенности протокола**

NFS работает по следующим принципам:

- **Стейтлессность** (в версиях до v4) - сервер не хранит состояние соединения
- **Прозрачность монтирования** - сетевые ресурсы выглядят как локальные
- **Кэширование** - ускорение доступа за счет локального хранения данных
- **Блокировка файлов** - координация доступа нескольких клиентов

**Безопасность и аутентификация в NFS**

Безопасность NFS развивалась вместе с версиями протокола:

- **NFSv2/v3** - базовая аутентификация через UID/GID
- **NFSv4** - интеграция с Kerberos, RPCSEC_GSS
- **NFSv4.1/4.2** - улучшенная безопасность, поддержка шифрования

**Сравнение версий NFS**

| Версия | Год | Особенности | Безопасность |
|--------|-----|-------------|--------------|
| NFSv2  | 1989| UDP, 32-битные файлы | Низкая |
| NFSv3  | 1995| TCP, 64-битные файлы | Средняя |
| NFSv4  | 2000| Состояние, Unicode | Высокая |
| NFSv4.1| 2010| pNFS, параллельный доступ | Высокая |
| NFSv4.2| 2016| Сжатие, улучшения | Очень высокая |

### Практическая часть NFS

**Установка и настройка NFS-сервера на ALT Linux**

1. **Обновление списка пакетов** - получение актуальной информации о доступных пакетах:
```bash
apt-get update
```

2. **Установка NFS-сервера** - установка основного пакета сервера и общих утилит:
```bash
apt-get install nfs-kernel-server nfs-common
```
- `nfs-kernel-server` - основной сервер NFS, реализованный в ядре
- `nfs-common` - общие утилиты и библиотеки для работы с NFS

3. **Создание каталога для экспорта** - подготовка директории, которую будем предоставлять клиентам:
```bash
mkdir -p /export/shared
chmod 755 /export/shared
chown nobody:nogroup /export/shared
```
- `mkdir -p` - создает директорию, включая промежуточные каталоги
- `chmod 755` - устанавливает права доступа: владелец - полный доступ, группа и другие - чтение и выполнение
- `chown nobody:nogroup` - устанавливает владельца и группу (стандартные для экспортируемых ресурсов)

4. **Настройка файла экспорта** - определение правил доступа к экспортируемым ресурсам:
```bash
# Формат: путь клиент(опции)
/export/shared 192.168.1.0/24(rw,sync,no_subtree_check)
```
- `/export/shared` - путь к экспортируемой директории
- `192.168.1.0/24` - сеть клиентов, которым разрешен доступ
- `rw` - разрешение на чтение и запись
- `sync` - синхронная запись (данные записываются на диск перед подтверждением)
- `no_subtree_check` - отключение проверки поддеревьев (улучшает производительность)

5. **Настройка автозапуска сервисов** - обеспечение автоматического запуска при загрузке системы:
```bash
systemctl enable rpcbind
systemctl enable nfs-server
```
- `rpcbind` - служба привязки RPC-портов (необходима для NFS)
- `nfs-server` - основной сервер NFS

6. **Запуск сервисов** - активация служб в текущей сессии:
```bash
systemctl start rpcbind
systemctl start nfs-server
```

7. **Применение конфигурации** - перечитывание файла экспорта и применение изменений:
```bash
exportfs -ra
```
- `-r` - перечитать файл экспорта

**Конфигурация exports**

Файл /etc/exports содержит правила экспорта файловых систем:

```bash
# Примеры конфигурации
/export/data    192.168.1.0/24(rw,sync,no_root_squash)
/export/backup  10.0.0.0/8(ro,sync,all_squash)
/export/home    *.example.com(rw,async,no_subtree_check)
```

**Настройка клиента NFS**

1. Установка клиентских пакетов:
```bash
apt-get install nfs-common
```

2. Создание точки монтирования:
```bash
mkdir -p /mnt/nfs-shared
```

3. Монтирование NFS-ресурса:
```bash
mount -t nfs server.example.com:/export/shared /mnt/nfs-shared
```

4. Автоматическое монтирование через /etc/fstab:
```bash
server.example.com:/export/shared /mnt/nfs-shared nfs rw,sync,hard,intr 0 0
```

**Примеры использования и мониторинга**

1. Проверка экспортированных ресурсов:
```bash
showmount -e localhost
exportfs -v
```

2. Мониторинг активных подключений:
```bash
nfsstat -c
nfsstat -s
```

3. Проверка состояния NFS:
```bash
rpcinfo -p
```

## Часть 2: SMB/CIFS (Server Message Block/Common Internet File System)

### Теоретические основы SMB/CIFS

**История и развитие протокола**

Протокол SMB имеет долгую историю развития:

- **1983** - Разработка Barry Feigenbaum в IBM
- **1984** - Реализация в IBM PC DOS
- **1990** - Microsoft включает SMB в Windows
- **1996** - Samba - реализация SMB для Unix-систем
- **2006** - SMB 2.0 - кардинальные улучшения
- **2012** - SMB 3.0 - шифрование, масштабируемость
- **2017** - SMB 3.1.1 - улучшения безопасности

**Архитектура и принципы работы**

Архитектура SMB/CIFS включает:

1. **Сетевой уровень** - транспортировка данных
2. **Сеансовый уровень** - управление соединениями
3. **Прикладной уровень** - файловые операции

Принципы работы:
- **Блокировка файлов** - предотвращение конфликтов доступа
- **Кэширование** - ускорение операций
- **Мультиплексирование** - несколько операций в одном соединении

**Особенности интеграции с Windows-средами**

SMB/CIFS идеально интегрируется с Windows:
- **Active Directory** - централизованная аутентификация
- **Групповые политики** - управление доступом
- **Резервное копирование** - VSS (Volume Shadow Copy Service)
- **Шифрование** - SMB 3.0 и выше

**Безопасность и аутентификация**

Механизмы безопасности SMB:
- **NTLM** - устаревший, но поддерживаемый
- **Kerberos** - современная аутентификация
- **Шифрование** - SMB 3.0+
- **Подпись сообщений** - защита от подделки

**Совместимость с различными ОС**

SMB/CIFS поддерживается на:
- Windows (все версии)
- Linux (через Samba)
- macOS (встроенная поддержка)
- FreeBSD, OpenBSD
- Android, iOS (через сторонние приложения)

### Практическая часть SMB/CIFS

**Установка и настройка Samba на ALT Linux**

1. Установка Samba:
```bash
apt-get update
apt-get install samba samba-common-bin
```

2. Создание каталога для общего доступа:
```bash
mkdir -p /srv/samba/shared
chmod 755 /srv/samba/shared
chown root:sambashare /srv/samba/shared
```

3. Создание группы для пользователей Samba:
```bash
groupadd sambashare
```

**Конфигурация smb.conf**

Файл /etc/samba/smb.conf - основной конфигурационный файл, определяющий поведение сервера:

```ini
[global]
    workgroup = WORKGROUP
    server string = Samba Server %v
    netbios name = ubuntu
    security = user
    map to guest = bad user
    dns proxy = no

[shared]
    path = /srv/samba/shared
    browsable = yes
    writable = yes
    guest ok = no
    read only = no
    create mask = 0755
    directory mask = 0755
    force user = nobody
```

**Пояснение параметров конфигурации:**

**Глобальные параметры [global]:**
- `workgroup` - имя рабочей группы (аналог домена в Windows)
- `server string` - описание сервера, отображаемое клиентам
- `netbios name` - имя сервера в сети NetBIOS
- `security = user` - режим безопасности: аутентификация по пользователю
- `map to guest` - политика обработки гостевых подключений
- `dns proxy` - отключение DNS-прокси (рекомендуется для производительности)

**Параметры общего ресурса [shared]:**
- `path` - путь к каталогу, предоставляемому для общего доступа
- `browsable = yes` - ресурс виден в списке общих ресурсов
- `writable = yes` - разрешена запись в ресурс
- `guest ok = no` - запрещен гостевой доступ (требуется аутентификация)
- `read only = no` - ресурс не только для чтения
- `create mask = 0755` - права доступа для создаваемых файлов
- `directory mask = 0755` - права доступа для создаваемых каталогов

**Создание общих ресурсов**

1. **Добавление пользователя Samba** - создание пользователя для доступа к общему ресурсу:
```bash
useradd -M -G sambashare testuser
smbpasswd -a testuser
```
- `useradd -M` - создает пользователя без домашней директории
- `-G sambashare` - добавляет пользователя в группу sambashare
- `smbpasswd -a testuser` - устанавливает пароль для пользователя Samba

2. **Настройка прав доступа** - установка владельца и прав доступа к общему ресурсу:
```bash
chown -R testuser:sambashare /srv/samba/shared
chmod -R 775 /srv/samba/shared
```
- `chown -R` - рекурсивно изменяет владельца и группу
- `chmod -R 775` - устанавливает права: владелец и группа - полный доступ, другие - чтение и выполнение

3. **Перезагрузка Samba** - применение изменений конфигурации:
```bash
systemctl restart smbd
systemctl enable smbd
```
- `systemctl restart smbd` - перезапускает демон Samba

**Настройка аутентификации и прав доступа**

1. Конфигурация аутентификации:
```ini
[global]
    security = user
    passdb backend = tdbsam
    obey pam restrictions = yes
    unix password sync = yes
```

2. Настройка прав доступа:
```ini
[shared]
    valid users = @sambashare
    write list = @sambashare
    read only = no
```

**Интеграция с доменом**

1. Присоединение к домену:
```bash
net ads join -U administrator
```

2. Настройка Kerberos:
```ini
[global]
    security = ads
    realm = EXAMPLE.COM
    workgroup = EXAMPLE
```

3. Настройка PAM:
```bash
authconfig --enablewinbind --enablewinbindauth --smbsecurity=ads \
    --smbworkgroup=EXAMPLE --smbrealm=EXAMPLE.COM \
    --enablewinbindusedefaultdomain --winbindtemplatehomedir=/home/%U \
    --winbindtemplateshell=/bin/bash --enablewinbindoffline \
    --winbindjoin=administrator --disablekrb5 --updateall
```

## Сравнение NFS и SMB/CIFS

### Архитектурные различия

| Характеристика | NFS | SMB/CIFS |
|----------------|-----|----------|
| Происхождение | Unix/Linux | Windows |
| Протокол | Stateless (v2/v3) | Stateful |
| Производительность | Высокая | Средняя |
| Безопасность | Средняя | Высокая |
| Интеграция | Unix-системы | Windows-системы |

### Сценарии использования

**NFS предпочтителен когда:**
- Работа в Unix/Linux среде
- Требуется высокая производительность
- Нужна простота настройки
- Используются одноранговые сети

**SMB/CIFS предпочтителен когда:**
- Работа в гетерогенной среде
- Требуется интеграция с Active Directory
- Нужна высокая безопасность
- Используются Windows-клиенты

### Преимущества и недостатки

**Преимущества NFS:**
- Высокая производительность
- Простота настройки
- Хорошая интеграция с Unix
- Поддержка большого количества клиентов

**Недостатки NFS:**
- Ограниченная безопасность (в старых версиях)
- Сложности с блокировкой файлов
- Проблемы с Unicode

**Преимущества SMB/CIFS:**
- Высокая безопасность
- Отличная интеграция с Windows
- Поддержка Unicode
- Гибкая система прав доступа

**Недостатки SMB/CIFS:**
- Ниже производительность
- Сложнее настройка
- Зависимость от Windows-инфраструктуры

## Устранение типичных проблем и ошибок

### Проблемы NFS

**Ошибка "Stale file handle"**
```bash
# Причина: сервер перезагрузился или изменил экспорт
# Решение: перезапустить клиент
umount /mnt/nfs-shared
mount -t nfs server:/export/shared /mnt/nfs-shared
```

**Проблема с монтированием**
```bash
# Проверка доступности сервера
ping server
# Проверка экспорта
showmount -e server
# Проверка портов
rpcinfo -p server
```

**Проблемы с производительностью**
```bash
# Использование async вместо sync
# Настройка размера блока
# Включение кэширования
```

### Проблемы SMB/CIFS

**Ошибка аутентификации**
```bash
# Проверка пользователя
pdbedit -L
# Проверка пароля
smbpasswd username
# Проверка конфигурации
testparm
```

**Проблемы с доступом**
```bash
# Проверка прав доступа
ls -la /srv/samba/shared
# Проверка конфигурации
smbstatus
# Проверка брандмауэра
firewall-cmd --list-services
```

**Проблемы с производительностью**
```bash
# Настройка параметров в smb.conf
socket options = TCP_NODELAY IPTOS_LOWDELAY
read raw = yes
write raw = yes
```

## Мониторинг и диагностика сетевых файловых систем

### Мониторинг NFS

**Инструменты мониторинга:**
```bash
# Статистика NFS
nfsstat -c  # клиентская статистика
nfsstat -s  # серверная статистика

# Активные подключения
netstat -an | grep :2049

# Производительность
iostat -x 1

# Логи
tail -f /var/log/syslog | grep nfs
```

**Показатели производительности:**
- Количество операций ввода-вывода
- Время отклика
- Использование пропускной способности
- Ошибки передачи

### Мониторинг SMB/CIFS

**Инструменты мониторинга:**
```bash
# Статус Samba
smbstatus

# Статистика
smbcontrol smbd status

# Логи
tail -f /var/log/samba/log.smbd

# Диагностика
smbclient -L localhost -U%
```

**Показатели производительности:**
- Количество активных сессий
- Объем переданных данных
- Время отклика
- Ошибки аутентификации

### Средства диагностики

**Общие инструменты:**
```bash

ping, traceroute, mtr
```

## Глоссарий терминов
- **Файловый сервер**: Сервер, предоставляющий доступ к файлам по сети
- **NFS**: Network File System - протокол сетевого доступа к файлам
- **SMB/CIFS**: Server Message Block/Common Internet File System
- **Samba**: Реализация SMB для Unix-систем
- **Экспорт**: Предоставление доступа к файловой системе
- **Монтирование**: Подключение удаленного ресурса к локальной файловой системе
