## Теоретическая часть

### 1. Архитектура управления процессами в Linux

В GNU/Linux процесс — это запущенная программа. Каждый процесс имеет:
- **PID** (Process ID) — уникальный идентификатор
- **PPID** (Parent PID) — идентификатор родительского процесса
- **UID/GID** — пользователь/группа, от имени которых работает процесс
- **Приоритет** (nice value) — значение от -20 (максимальный приоритет) до +19 (минимальный)
- **Состояние**: R (running), S (sleeping), D (uninterruptible sleep), Z (zombie), T (stopped)

**Команды просмотра процессов:**
| Команда | Назначение | Особенности |
|---------|------------|-------------|
| `ps aux` | Полный список всех процессов | `a` — все терминалы, `u` — детальный, `x` — без терминала |
| `ps -ef` | Альтернативный формат (System V) | Подходит для скриптов |
| `top` / `htop` | Интерактивный мониторинг | Обновляется в реальном времени |
| `pgrep` | Поиск PID по имени | `pgrep -l bash` |
| `pidof` | Получение PID по имени процесса | `pidof nginx` |

### 2. Управление жизненным циклом процессов

**Завершение процессов:**
| Команда | Сигнал | Назначение |
|---------|--------|------------|
| `kill <PID>` | TERM (15) | Вежливое завершение (default) |
| `kill -9 <PID>` | KILL (9) | Принудительное завершение (нельзя игнорировать) |
| `kill -15 <PID>` | TERM | Явное указание graceful termination |
| `pkill <имя>` | TERM | Завершение по имени процесса |
| `pkill -9 <имя>` | KILL | Принудительное завершение по имени |
| `killall <имя>` | TERM | Завершение всех процессов с именем |
| `kill -1 <PID>` | HUP | Перезагрузка конфигурации (для демонов) |

**Изменение приоритета:**
| Команда | Назначение |
|---------|------------|
| `nice -n <N> команда` | Запуск процесса с приоритетом N (только root может использовать отрицательные значения) |
| `renice -n <N> -p <PID>` | Изменение приоритета уже запущенного процесса |
| `nice` (без параметров) | Вывод текущего приоритета |

### 3. Системные службы (daemons) и systemd

**systemd** — современная система инициализации в Linux, управляющая службами (демонами).

**Unit-файлы systemd:**
**Unit** — это концептуальная единица управления в systemd. Unit-файлы описывают, как systemd должен управлять различными ресурсами: службами (service), дисками (mount), таймерами (timer), сокетами (socket) и т.д.

**Типы unit-файлов:**
| Тип | Расширение | Назначение |
|-----|------------|------------|
| Служба | `.service` | Запуск и управление демонами/процессами |
| Таймер | `.timer` | Запуск служб по расписанию (аналог cron) |
| Сокет | `.socket` | Активация службы при подключении к сокету |
| Mount | `.mount` | Монтирование файловых систем |
| Swap | `.swap` | Управление swap-пространством |
| Target | `.target` | Группировка unit (аналог runlevels) |
| Path | `.path` | Активация при изменении файла/директории |

**Расположение unit-файлов:**
- `/etc/systemd/system/` — административные unit-файлы (создаваемые вручную)
- `/run/systemd/system/` — временные unit-файлы (во время сессии)
- `/usr/lib/systemd/system/` — unit-файлы, поставляемые пакетами

**Структура unit-файла службы:**
```ini
[Unit]
Description=Описание службы
Documentation=man:service(8)
After=network.target  # Запуск после указанных unit
Requires=network.target  # Зависимости
Wants=network.target   # Слабые зависимости

[Service]
Type=simple  # Тип службы: simple, forking, oneshot, dbus, notify, idle
ExecStart=/usr/bin/daemon  # Команда запуска
ExecStop=/bin/kill $MAINPID  # Команда остановки
ExecReload=/bin/kill -HUP $MAINPID  # Перезагрузка конфига
Restart=on-failure  # Перезапуск при ошибке
RestartSec=5  # Задержка перед перезапуском
User=nobody  # Пользователь, от имени которого работает
Group=nogroup  # Группа

[Install]
WantedBy=multi-user.target  # В каком target включить службу
```

**Основные типы служб (Type=):**
- `simple` (по умолчанию) — процесс запускается и работает в foreground
- `forking` — процесс делает fork, родитель завершается (демонизация)
- `oneshot` — процесс выполняется один раз и завершается
- `dbus` — служба активируется через D-Bus
- `notify` — служба уведомляет systemd о готовности через sd_notify()
- `idle` — служба запускается, когда система простаивает

**Управление unit-файлами:**
```bash
# Создание/редактирование unit-файла
sudo nano /etc/systemd/system/hello.service

# Проверка синтаксиса unit-файла
sudo systemctl daemon-reload
sudo systemctl status hello.service

# Загрузка нового unit
sudo systemctl daemon-reload

# Включение/отключение автозапуска
sudo systemctl enable hello.service
sudo systemctl disable hello.service

# Просмотр зависимостей unit
systemctl list-dependencies multi-user.target

# Просмотр всех unit (фильтрация)
systemctl list-units --type=service --state=running
systemctl list-units --type=service --all
```

**Создание собственной службы — practice example:**
1. Создайте скрипт `/usr/local/bin/myscript.sh` с нужной логикой
2. Создайте unit-файл `/etc/systemd/system/myservice.service` с appropriate настройками
3. Выполните `systemctl daemon-reload`
4. Запустите `systemctl start myservice`
5. Проверьте статус `systemctl status myservice`
6. Включите автозагрузку `systemctl enable myservice`

**Анализ unit-файлов:**
```bash
# Показать полную конфигурацию unit (включая defaults)
systemctl show sshd.service

# Показать конкретный параметр
systemctl show sshd.service -p ExecStart

# Экспорт конфигурации в видеINI
systemctl show sshd.service --property=All > sshd.conf
```

**Перезагрузка конфигурации vs Перезапуск службы:**
- `systemctl reload <служба>` — перезагружает конфигурацию без прерывания работы (если служба поддерживает SIGHUP)
- `systemctl restart <служба>` — полная остановка и повторный запуск службы

**Основные команды systemctl:**

| Действие | Команда | Пример |
|----------|---------|--------|
| Статус службы | `systemctl status <служба>` | `systemctl status sshd` |
| Запуск | `systemctl start <служба>` | `systemctl start cron` |
| Остановка | `systemctl stop <служба>` | `systemctl stop cron` |
| Перезапуск | `systemctl restart <служба>` | `systemctl restart nginx` |
| Перезагрузка конфига | `systemctl reload <служба>` | `systemctl reload nginx` |
| Включение автозапуска | `systemctl enable <служба>` | `systemctl enable sshd` |
| Отключение автозапуска | `systemctl disable <служба>` | `systemctl disable cron` |
| Просмотр всех служб | `systemctl list-units --type=service` | — |
| Просмотр автозагрузки | `systemctl is-enabled <служба>` | — |
| Принудительная остановка | `systemctl kill <служба>` | — |

**Состояния служб:**
- `active (running)` — служба запущена и работает
- `active (exited)` — служба успешно завершила работу (одноразовая)
- `inactive` — служба остановлена
- `failed` — служба завершилась с ошибкой
- `activating` — служба запускается
- `deactivating` — служба останавливается

**Логи служб:**
```bash
journalctl -u <служба>        # Логи конкретной службы
journalctl -u <служба> -f     # Отслеживание логов в реальном времени
journalctl -u <служба> --since "1 hour ago"
```

```bash
echo "Текущее время: $(date)"
echo
echo "Топ-5 процессов по использованию CPU:"
ps aux --sort=-%cpu | head -6
echo

ps aux --sort=-%mem | head -6
echo

```

2. Запустите: `./monitor.sh`
3. Добавьте вывод в отчёт.

### Задание B: Анализ zombie-процессов
1. Создайте простой zombie-процесс:
   ```bash
   ( while true; do ( (sleep 1;) & wait; ) done ) &
   ```
2. Найдите zombie-процессы:
   ```bash
   ps aux | grep '^.* Z .*$'
   ```
3. Завершите родительский процесс zombie.
4. Объясните, что такое zombie и как он образуется.

### Задание C: Управление службой через systemd
1. Создайте простой bash-скрипт `/usr/local/bin/hello.sh`:
   ```bash
   #!/bin/bash
   while true; do
     echo "Служба работает: $(date)" >> /tmp/hello.log
     sleep 30
   done
   ```
   Сделайте исполняемым: `chmod +x /usr/local/bin/hello.sh`.

2. Создайте unit-файл службы `/etc/systemd/system/hello.service`:
   ```ini
   [Unit]
   Description=Test Hello Service
   After=network.target

   [Service]
   Type=simple
   ExecStart=/usr/local/bin/hello.sh
   Restart=on-failure

   [Install]
   WantedBy=multi-user.target
   ```

3. Загрузите службу в systemd:
   ```bash
   sudo systemctl daemon-reload
   ```

4. Запустите и включите автозагрузку:
   ```bash
   sudo systemctl start hello
   sudo systemctl enable hello
   ```

5. Проверьте статус и логи:
   ```bash
   systemctl status hello
   journalctl -u hello -f
   ```

6. Остановите и удалите службу:
   ```bash
   sudo systemctl stop hello
   sudo systemctl disable hello
   sudo rm /etc/systemd/system/hello.service
   sudo systemctl daemon-reload
   ```

## Возможные ошибки и способы устранения

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `bash: kill: (PID) - Permission denied` | Нет прав root на завершение чужого процесса | Использовать `sudo kill` или войти как root. Некоторые системные процессы защищены от завершения обычными пользователями. |
| `bash: renice: Permission denied` | Попытка изменить nice без прав (уменьшить nice, т.е. повысить приоритет) | Обычный пользователь может только увеличивать nice (понижать приоритет). Для уменьшения нужен root. |
| `bash: systemctl: command not found` | systemd не установлен или не в PATH | ALT Linux использует systemd по умолчанию. Проверить: `which systemctl`. Если нет — установить `systemd-sysv`. |
| `Failed to stop <служба>.service: Unit <служба>.service not loaded.` | Служба не установлена или отключена | Проверить существование: `systemctl list-unit-files | grep служба`. Если службы нет в списке, возможно, она не установлена; выберите другую службу из списка доступных. |
| `Job for <служба>.service failed because the control process exited with error code.` | Ошибка в unit-файле или исполняемом файле | Проверить логи: `journalctl -u служба -xe`. |
| `No such file or directory` при создании unit-файла | Неправильный путь или нет прав | Убедиться в правильности пути `/etc/systemd/system/` и наличии прав `sudo`. |
| `Zombie processes appear in ps output` | Родительский процесс не вызвал `wait()` | Zombie не потребляет ресурсов (только запись в таблице процессов). Завершить родительский процесс или дождаться. |
| `ps: unknown option: p` | Неправильный синтаксис команды | Проверить синтаксис. В некоторых дистрибутивах `ps -p` работает, в других `ps --pid`. |
| `free показывает 0 swap` | Swap не настроен (нет раздела/файла) | Это нормально для некоторых конфигураций. Пропустить задание, если swap действительно отсутствует. |
| `bash: sudo: command not found` | Нет sudo в системе | Войти как root: `su -`, установить sudo: `apt-get install sudo`. |

## Полезные команды (шпаргалка)

```bash
# Процессы
ps aux                              # все процессы
ps -ef                              # альтернативный формат
ps -p PID -o pid,user,cmd          # конкретный процесс
pstree -p                          # дерево процессов с PID
pgrep -l имя_процесса              # поиск PID по имени
pidof имя_процесса                  # получить PID

# Завершение
kill PID                            # SIGTERM
kill -9 PID                         # SIGKILL
pkill имя_процесса                  # по имени
killall имя_процесса                # все процессы с именем

# Приоритеты
nice -n 10 команда &                # запуск с nice 10
renice -n 5 -p PID                  # изменить nice
nice                                # показать текущий nice

# systemd
systemctl status служба             # статус
systemctl start/stop служба         # запуск/остановка
systemctl restart служба            # перезапуск
systemctl enable/disable служба     # автозагрузка
systemctl is-active служба          # активна ли?
systemctl is-enabled служба         # включена ли?
journalctl -u служба                # логи службы
journalctl -u служба -f             # логи в реальном времени
systemctl list-units --type=service # все службы
systemctl --type=service --state=failed # failed службы

# Ресурсы
top                                # мониторинг
htop                               # улучшенный мониторинг
free -h                            # память
df -h                              # диски
```

## Примечания по специфике ALT Linux

1. В ALT Linux используется **systemd** как система инициализации по умолчанию (начиная с версии 7 и выше).
2. Для управления пакетами используется `apt-get` (гибридный `apt-rpm`), но это не относится к процессам/службам.
3. Команды `ps`, `kill`, `systemctl` — стандартные GNU/Linux, идентичны другим дистрибутивам.
4. Если `systemctl` недоступен, возможно используется старая система инициализации (SysV). Но в ALT Linux p11 systemd стандартен.
5. Для просмотра журналов вместо `tail -f /var/log/syslog` используйте `journalctl`.
6. Для работы с `journalctl` может потребоваться вхождение в группу `systemd-journal` или использование `sudo`.
