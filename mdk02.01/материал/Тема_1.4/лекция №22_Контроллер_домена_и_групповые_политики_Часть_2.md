## 1. Контроллер домена: понятие, функции и архитектура

### 1.1. Понятие контроллера домена

**Контроллер домена (Domain Controller, DC)** — это сервер, который отвечает за управление безопасностью и аутентификацией в доменной инфраструктуре. Контроллер домена хранит базу данных каталога Active Directory и обрабатывает запросы аутентификации от пользователей и компьютеров.

**Ключевые функции контроллера домена:**
- **Аутентификация пользователей** — проверка подлинности учетных данных
- **Авторизация доступа** — контроль доступа к сетевым ресурсам
- **Хранение каталога** — хранение информации об объектах домена
- **Репликация данных** — синхронизация с другими контроллерами домена
- **Применение политик** — обработка и применение групповых политик
- **Управление доверием** — поддержка доверительных отношений между доменами

### 1.2. Архитектура контроллера домена Samba DC

**Компоненты Samba DC:**

```
Samba Domain Controller
├── LDAP сервер (для хранения объектов каталога)
├── Kerberos KDC (для аутентификации)
├── DNS сервер (для разрешения имен)
├── SMB/CIFS сервер (для доступа к файлам)
├── Winbind (для интеграции с Unix-системами)
├── NTP сервер (для синхронизации времени)
└── Служба групповых политик (GPO)
```

**База данных каталога:**
- Хранится в формате LDAP
- Содержит информацию о пользователях, группах, компьютерах, OU, GPO
- Реплицируется между всеми контроллерами домена
- В Samba DC используется встроенная база данных (tdb/ldb)

### 1.3. Роли контроллера домена

**Основные роли (FSMO):**

1. **Schema Master (Мастер схемы)**
   - Единственный контроллер, который может вносить изменения в схему
   - В Samba DC: определяется автоматически

2. **Domain Naming Master (Мастер имен доменов)**
   - Управляет добавлением/удалением доменов в лесу
   - В Samba DC: обычно тот же, что и Schema Master

3. **PDC Emulator (Эмулятор PDC)**
   - Обрабатывает изменения паролей
   - Является источником точного времени
   - В Samba DC: критически важная роль

4. **RID Master (Мастер RID)**
   - Распределяет пулы относительных идентификаторов
   - В Samba DC: автоматическое управление

5. **Infrastructure Master (Мастер инфраструктуры)**
   - Обновляет ссылки на объекты из других доменов
   - В Samba DC: не требуется в однодоменной среде

**Проверка ролей FSMO в Samba:**
```bash
# Просмотр владельцев ролей
samba-tool fsmo show

# Пример вывода:
# Schema FSMO Owner: CN=Samba,CN=Schema,CN=Directory Service,CN=Windows NT,CN=Services,CN=Configuration,DC=company,DC=local
# Infrastructure FSMO Owner: CN=Samba,CN=Infrastructure,DC=company,DC=local
# PDC FSMO Owner: CN=Samba,CN=System,DC=company,DC=local
# RID FSMO Owner: CN=Samba,CN=System,DC=company,DC=local
# Domain Naming FSMO Owner: CN=Samba,CN=Partitions,CN=Configuration,DC=company,DC=local
```

## 2. Роль контроллера домена в инфраструктуре Samba DC

### 2.1. Взаимодействие с клиентами

**Процесс аутентификации:**
```
1. Клиент отправляет запрос на вход
2. Контроллер домена проверяет учетные данные через Kerberos
3. При успехе выдается билет (ticket)
4. Клиент использует билет для доступа к ресурсам
```

**Пример входа в домен:**
```bash
# На клиенте Linux
kinit administrator@COMPANY.LOCAL
# Ввод пароля
# Получение билета Kerberos

klist
# Проверка полученных билетов
```

### 2.2. Репликация между контроллерами домена

**Типы репликации:**
- **Многомастер-репликация** — все контроллеры равноправны
- **Одномастер-репликация** — для операций FSMO

**Настройка репликации в Samba:**
```bash
# Проверка репликации
samba-tool drs showrepl

# Принудительная репликация
samba-tool drs replicate DC1 DC2 --full
```

### 2.3. Службы контроллера домена

**Основные службы Samba DC:**
- `samba-ad-dc` — основная служба Active Directory
- `samba` — файловая служба (если включена)
- `winbind` — интеграция с Unix-системами

**Управление службами в ALT Linux:**
```bash
# Проверка статуса
systemctl status samba-ad-dc

# Перезапуск службы
systemctl restart samba-ad-dc

# Включение автозагрузки
systemctl enable samba-ad-dc
```

## 3. ADMX-шаблоны: структура, назначение и использование

### 3.1. Понятие ADMX-шаблонов

**ADMX (Administrative Template XML)** — это XML-файлы, которые определяют настройки реестра, управляемые групповыми политиками. ADMX-шаблоны предоставляют графический интерфейс для настройки политик в редакторе групповых политик.

**Отличие ADMX от ADM:**
- **ADM** — устаревший формат (текстовый)
- **ADMX** — современный формат (XML), поддерживает локализацию

**Структура ADMX-файла:**
```xml
<?xml version="1.0" encoding="UTF-8"?>
<policyDefinitions revision="1.0" schemaVersion="1.0">
  <policyNamespaces>
    <target prefix="example" namespace="Microsoft.Policies.example"/>
    <using prefix="inetres" namespace="Microsoft.Policies.InternetExplorer"/>
  </policyNamespaces>
  
  <resources minRequired="1">
    <stringTable>
      <string id="example_policy">Пример политики</string>
      <string id="example_help">Описание политики</string>
    </stringTable>
  </resources>
  
  <categories>
    <category name="example_category">
      <displayName>
        <string>Пример категории</string>
      </displayName>
    </category>
  </categories>
  
  <policies>
    <policy name="example_setting" class="Both" 
            displayName="$(string.example_policy)" 
            explain="$(string.example_help)" 
            presentation="$(presentation.example_setting)">
      <parentCategory ref="example_category"/>
      <supportedOn ref="windows:SUPPORTED_Windows7"/>
      <keywords>
        <keyword>example</keyword>
      </keywords>
      <elements>
        <decimalTextBox id="example_value" defaultValue="100"/>
      </elements>
    </policy>
  </policies>
</policyDefinitions>
```

### 3.2. Типы ADMX-шаблонов

**Встроенные шаблоны Windows:**
- `windows.admx` — основные настройки Windows
- `inetres.admx` — настройки Internet Explorer
- `system.admx` — системные настройки
- `network.admx` — сетевые настройки

**Сторонние шаблоны:**
- Шаблоны Microsoft Office
- Шаблоны для браузеров (Chrome, Firefox)
- Шаблоны для антивирусного ПО
- Пользовательские шаблоны организаций

### 3.3. Структура ADMX-файлов

**Основные элементы:**
1. **policyDefinitions** — корневой элемент
2. **policyNamespaces** — пространства имен
3. **resources** — ресурсы (строки, описания)
4. **categories** — категории политик
5. **policies** — определения политик

**Пример простой политики:**
```xml
<policy name="DisableControlPanel" 
        class="User" 
        displayName="$(string.DisableCP)" 
        explain="$(string.DisableCP_Help)">
  <parentCategory ref="ControlPanel"/>
  <supportedOn ref="windows:SUPPORTED_WindowsXP"/>
  <elements>
    <boolean id="DisableCP" valueName="NoControlPanel" 
             trueValue="1" falseValue="0"/>
  </elements>
</policy>
```

### 3.4. Локализация ADMX-шаблонов

**Файлы ADML (Administrative Template Language):**
- Содержат локализованные строки
- Расположены в папках с кодом языка (ru-RU, en-US)
- Ссылаются на ADMX-файлы

**Структура локализованного файла:**
```
Central Store\
├── PolicyDefinitions\
│   ├── windows.admx
│   ├── system.admx
│   └── ...
└── ru-RU\
    ├── windows.adml
    ├── system.adml
    └── ...
```

## 4. Настройка центрального хранилища ADMX в Samba DC

### 4.1. Понятие центрального хранилища

**Центральное хранилище (Central Store)** — это сетевая папка SYSVOL, в которой хранятся ADMX-файлы, доступные всем контроллерам домена и администраторам.

**Преимущества центрального хранилища:**
- Единая точка управления шаблонами
- Автоматическая репликация между DC
- Доступность для всех администраторов
- Согласованность настроек

### 4.2. Создание центрального хранилища в Samba DC

**Шаг 1: Создание структуры каталогов**
```bash
# Переход в папку SYSVOL
cd /var/lib/samba/sysvol/company.local/Policies

# Создание папки PolicyDefinitions
mkdir -p PolicyDefinitions/ru-RU
```

**Шаг 2: Копирование ADMX-файлов**
```bash
# Копирование с Windows-системы
# На Windows: C:\Windows\PolicyDefinitions\

# Копирование в Samba DC
cp -r /mnt/windows/PolicyDefinitions/* /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions/

# Копирование локализованных файлов
cp -r /mnt/windows/PolicyDefinitions/ru-RU/* /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions/ru-RU/
```

**Шаг 3: Настройка прав доступа**
```bash
# Установка правильных прав
chown -R root:root /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions
chmod -R 755 /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions
```

### 4.3. Добавление пользовательских ADMX-шаблонов

**Пример добавления шаблона Chrome:**
```bash
# Загрузка шаблона Chrome
wget https://dl.google.com/dl/chrome/install/ChromeADMX.msi

# Извлечение ADMX-файлов
msiextract ChromeADMX.msi -C chrome_admx

# Копирование в центральное хранилище
cp chrome_admx/*.admx /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions/
cp -r chrome_admx/ru-RU /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions/
```

### 4.4. Проверка центрального хранилища

**Проверка структуры:**
```bash
# Просмотр содержимого
ls -la /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions/

# Проверка локализованных файлов
ls -la /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions/ru-RU/
```

**Проверка репликации:**
```bash
# На другом контроллере домена
ls -la /var/lib/samba/sysvol/company.local/Policies/PolicyDefinitions/

# Проверка доступности по сети
ls -la \\company.local\SYSVOL\company.local\Policies\PolicyDefinitions\
```

## 5. Инструмент ADMC (Admin Center) для Linux: возможности и установка

### 5.1. Понятие ADMC

**ADMC (Admin Center)** — это кроссплатформенный графический инструмент для администрирования Active Directory, разработанный компанией Symantec (ранее Quest Software). ADMC предоставляет удобный интерфейс for управления объектами домена, групповыми политиками и другими компонентами Active Directory.

**Ключевые возможности ADMC:**
- Управление пользователями, группами, компьютерами
- Управление подразделениями (OU)
- Редактирование групповых политик (GPO)
- Управление доверительными отношениями
- Мониторинг репликации
- Поиск и фильтрация объектов
- Массовые операции

### 5.2. Установка ADMC в ALT Linux

**Шаг 1: Подготовка системы**
```bash
# Обновление системы
apt-get update

# Установка зависимостей
apt-get install openjdk-11-jre wget
```

**Шаг 2: Загрузка ADMC**
```bash
# Загрузка последней версии
wget https://github.com/Computershare/admc/releases/download/v7.1.0/admc-7.1.0-linux-x64.run

# Или с официального сайта
wget https://admc.io/download/admc-linux-x64.run
```

**Шаг 3: Установка**
```bash
# Предоставление прав на выполнение
chmod +x admc-7.1.0-linux-x64.run

# Запуск установщика
./admc-7.1.0-linux-x64.run

# Или установка в тихом режиме
./admc-7.1.0-linux-x64.run --mode unattended
```

**Шаг 4: Запуск ADMC**
```bash
# Запуск из командной строки
admc

# Или через меню приложений
# Приложения → Администрирование → ADMC
```

### 5.3. Настройка подключения к домену

**Подключение к контроллеру домена:**
1. Запустите ADMC
2. Выберите "File" → "Connect"
3. Введите параметры:
   - **Server**: имя или IP контроллера домена
   - **Port**: 389 (LDAP) или 636 (LDAPS)
   - **Username**: домен\администратор
   - **Password**: пароль администратора
4. Нажмите "Connect"

**Пример подключения через командную строку:**
```bash
admc --server=dc1.company.local --user=administrator --password=***
```

### 5.4. Интерфейs ADMC

**Основные компоненты интерфейса:**
- **Дерево консоли** — иерархия домена
- **Панель объектов** — список объектов выбранного типа
- **Панель свойств** — свойства выбранного объекта
- **Панель действий** — доступные операции
- **Журнал** — лог операций

**Горячие клавиши:**
- `F5` — обновить
- `F3` — найти объект
- `Ctrl+N` — создать объект
- `Ctrl+D` — удалить объект
- `Alt+Enter` — свойства объекта

## 6. Управление доменом через ADMC

### 6.1. Управление пользователями

**Создание пользователя:**
1. В дереве консоли выберите подразделение (OU)
2. Правая кнопка → "New" → "User"
3. Заполните поля:
   - Имя, фамилия
   - Логин (User logon name)
   - Пароль
4. Нажмите "OK"

**Массовое создание пользователей:**
```bash
# Импорт из CSV-файла
# Файл users.csv:
# name,surname,login,password,ou
# Иван,Иванов,iivanov,Password123,OU=Users,DC=company,DC=local

# Через командную строку ADMC
admc import-csv --file=users.csv --server=dc1.company.local
```

**Изменение свойств пользователя:**
- Правая кнопка на пользователе → "Properties"
- Вкладка "Account" — настройки учетной записи
- Вкладка "Member Of" — членство в группах
- Вкладка "Profile" — профиль и домашняя папка

### 6.2. Управление группами

**Создание группы:**
1. Выберите OU
2. Правая кнопка → "New" → "Group"
3. Укажите:
   - Имя группы
   - Тип группы (Security/Distribution)
   - Область действия (Global/Domain Local/Universal)
4. Нажмите "OK"

**Добавление участников в группу:**
1. Откройте свойства группы
2. Вкладка "Members"
3. Нажмите "Add"
4. Выберите пользователей или другие группы
5. Нажмите "OK"

### 6.3. Управление подразделениями (OU)

**Создание OU:**
1. Выберите домен или родительское OU
2. Правая кнопка → "New" → "Organizational Unit"
3. Введите имя OU
4. Нажмите "OK"

**Делегирование прав на OU:**
1. Откройте свойства OU
2. Вкладка "Security"
3. Нажмите "Advanced"
4. Добавьте пользователя или группу
5. Назначьте разрешения

### 6.4. Управление групповыми политиками через ADMC

**Создание GPO:**
1. В дереве консоли разверните "Group Policy Objects"
2. Правая кнопка → "New"
3. Введите имя GPO
4. Нажмите "OK"

**Редактирование GPO:**
1. Правая кнопка на GPO → "Edit"
2. Откроется редактор групповых политик
3. Настройте нужные параметры
4. Закройте редактор

**Привязка GPO к OU:**
1. Выберите OU
2. Вкладка "Group Policy"
3. Нажмите "Add"
4. Выберите GPO
5. Настройте порядок применения

### 6.5. Поиск и фильтрация объектов

**Базовый поиск:**
- Нажмите `F3` или "Edit" → "Find"
- Выберите тип объекта
- Введите критерии поиска
- Нажмите "Find Now"

**Расширенный поиск (LDAP-фильтр):**
```ldap
# Поиск всех пользователей в OU Sales
(&(objectClass=user)(objectCategory=person)(ou=Sales))

# Поиск заблокированных пользователей
(&(objectClass=user)(userAccountControl:1.2.840.113556.1.4.803:=2))

# Поиск пользователей с истекающим паролем
(&(objectClass=user)(passwordLastSet<=20240101000000.0Z))
```

## 7. Ввод клиентских машин в домен

### 7.1. Подготовка клиентской машины

**Требования к клиенту:**
- Настроенный DNS (должен указывать на контроллер домена)
- Синхронизированное время (разница не более 5 минут)
- Сетевая связность с контроллером домена

**Проверка готовности:**
```bash
# Проверка DNS
nslookup company.local

# Проверка времени
ntpdate -q dc1.company.local

# Проверка связности
ping dc1.company.local
```

### 7.2. Ввод в домен через realmd (ALT Linux)

**Шаг 1: Установка необходимых пакетов**
```bash
# Установка realmd и зависимостей
apt-get install realmd sssd sssd-tools adcli krb5-user

# Проверка доступности домена
realm discover company.local
```

**Шаг 2: Ввод в домен**
```bash
# Ввод компьютера в домен
realm join company.local --user=administrator

# Или с указанием конкретного контроллера
realm join dc1.company.local --user=administrator
```

**Шаг 3: Настройка SSSD**
```bash
# Проверка конфигурации SSSD
cat /etc/sssd/sssd.conf

# Перезапуск службы
systemctl restart sssd

# Включение автозагрузки
systemctl enable sssd
```

**Шаг 4: Проверка входа**
```bash
# Проверка информации о домене
realm list

# Проверка пользователей домена
id user@company.local

# Вход под доменным пользователем
su - user@company.local
```

### 7.3. Ввод в домен Windows-клиента

**Через графический интерфейс:**
1. "Панель управления" → "Система"
2. "Изменить параметры" → "Изменить"
3. Выберите "Является членом домена"
4. Введите имя домена: `company.local`
5. Введите учетные данные администратора домена
6. Перезагрузите компьютер

**Через PowerShell:**
```powershell
# Ввод в домен
Add-Computer -DomainName "company.local" -Credential "administrator@company.local" -Restart

# Проверка подключения
Test-ComputerSecureChannel -Verbose
```

### 7.4. Настройка политик для клиентских машин

**Применение политик после ввода в домен:**
```bash
# На Linux клиенте
# Политики применяются автоматически при входе

# Принудительное обновление
/usr/sbin/gpupdate --force

# Проверка примененных политик
/usr/sbin/gpresult --verbose
```

**На Windows клиенте:**
```powershell
# Обновление политик
gpupdate /force

# Просмотр примененных политик
gpresult /r
```

## 8. Практическое применение групповых политик

### 8.1. Создание базовой политики безопасности

**Пример: Политика минимальной длины пароля**

1. Откройте ADMC
2. Разверните "Group Policy Objects"
3. Создайте новую GPO: "Password Policy"
4. Отредактируйте GPO:
   - Computer Configuration → Policies → Windows Settings → Security Settings → Account Policies → Password Policy
   - "Minimum password length": 12 символов
   - "Password must meet complexity requirements": Enabled
   - "Minimum password age": 1 day
   - "Maximum password age": 90 days
5. Привяжите GPO to домен

**Применение политики:**
```bash
# На контроллере домена
samba-tool gpo setlink "DC=company,DC=local" "Password Policy"

# Принудительное применение
samba-tool gpo setlinkstate "DC=company,DC=local" 1 --enforced=yes
```

### 8.2. Настройка политики блокировки учетных записей

**Пример: Политика блокировки после неудачных попыток**

1. В ADMC создайте GPO: "Account Lockout Policy"
2. Настройте:
   - "Account lockout threshold": 5 invalid logon attempts
   - "Account lockout duration": 30 minutes
   - "Reset account lockout counter after": 30 minutes
3. Привяжите GPO to домен

### 8.3. Создание политики для рабочих станций

**Пример: Ограничение доступа к панели управления**

1. Создайте GPO: "Workstation Restrictions"
2. Настройте:
   - User Configuration → Policies → Administrative Templates → Control Panel
   - "Prohibit access to Control Panel and PC settings": Enabled
3. Привяжите GPO to OU "Workstations"

### 8.4. Развертывание обоев рабочего стола

**Пример: Корпоративные обои**

1. Подготовьте изображение (например, `wallpaper.jpg`)
2. Поместите в сетевую папку: `\\company.local\SYSVOL\company.local\wallpaper\wallpaper.jpg`
3. Создайте GPO: "Corporate Wallpaper"
4. Настройте:
   - User Configuration → Policies → Administrative Templates → Desktop → Desktop
   - "Desktop Wallpaper": `\\company.local\SYSVOL\company.local\wallpaper\wallpaper.jpg`
   - "Wallpaper Style": Fill
5. Привяжите GPO to нужные OU

### 8.5. Настройка перенаправления папок

**Пример: Перенаправление папки "Документы"**

1. Создайте GPO: "Folder Redirection"
2. Настройте:
   - User Configuration → Policies → Windows Settings → Folder Redirection → Documents
   - Setting: "Basic - Redirect everyone's folder to the same location"
   - Target folder location: `\\server\users$\%USERNAME%\Documents`
   - Settings: "Grant the user exclusive rights to Documents"
3. Привяжите GPO to OU "Users"
